# Журнал изменений

## v0.1.19 - 2026-04-17

### Улучшено

- Поведение `Edit Message` приведено к реальному контракту Max API: `message_id` снова отправляется в query-параметре `PUT /messages?message_id=...`. Локальная живая проверка с тем же credential из n8n показала, что query-вариант возвращает `200`, а body-only вариант — `400` с `proto.payload`.
- Сообщения об ошибках для `Edit Message` стали точнее: если Max API вернёт вложенный JSON с причиной ошибки, нода теперь показывает её пользователю вместо общего `Request failed with status code 400`.

### Для разработки

- Добавлен регрессионный тест на извлечение текста ошибки из `response.data.message`.
- Обновлены тесты `GenericFunctions.test.ts` под подтверждённый live-контракт `message_id` в query.

### Кому важно

- Пользователям, у которых `Edit Message` сейчас падает с `400` после релиза `v0.1.18`.
- Пользователям, которым нужна понятная диагностика ошибок Max API без ручной проверки через `HTTP Request`.

### Что проверить после обновления

- В `Max` → `Edit Message` повторить сценарий `Send Message` -> `Edit Message` на свежем сообщении и убедиться, что редактирование проходит успешно.
- Если редактирование всё ещё не проходит, проверить новый текст ошибки: он должен содержать причину из ответа Max API.
- Проверить сценарии `Text Format = Plain Text`, `HTML` и `Markdown`.

## v0.1.18 - 2026-04-17

### Улучшено

### Кому важно

- Пользователям, у которых `Edit Message` падал из-за расхождения между ожидаемым контрактом API и поведением ноды.
- Пользователям, которые хотят предсказуемое поведение `Edit Message` без переключения между разными схемами запроса.

### Что проверить после обновления

- Проверить, что `Delete Message` и `Send Message` работают без изменений в текущем workflow.

## v0.1.17 - 2026-04-17

### Улучшено

- Исправлено редактирование сообщений в `Max` node: при `Edit Message` нода сначала использует контракт `PUT /messages?message_id=...`, а при ответе `404` автоматически повторяет запрос с `message_id` в JSON body. Это закрывает кейс из issue `#11`, где запрос в `HTTP Request` node проходил, а в `Max` node — нет.

### Кому важно

- Пользователям workflow, у которых `Edit Message` падал с `404`, хотя тот же вызов через `HTTP Request` node выполнялся успешно.
- Пользователям, которые редактируют сообщения в разных API-контурах Max, где может отличаться ожидаемое расположение `message_id`.

### Что проверить после обновления

- В `Max` → `Edit Message` отредактировать свежее сообщение и убедиться, что операция проходит без `404`.
- Повторить редактирование с `Text Format = Markdown` и проверить, что при ошибке неподдерживаемой разметки сообщение отправляется в plain text fallback.
- Проверить, что `Send Message` и `Delete Message` работают как раньше с теми же credentials.
